# Data visualization

How OCHA turns humanitarian data into clear, accurate, accessible visuals. These rules come from 10+ years of OCHA's **VIZCOM** training (700+ staff trained) and are codified in the [Humanitarian Dataviz Tool](https://brand.unocha.org) used daily by the Brand and Design Unit.

  <p style={{ margin: '6px 0 0', fontSize: '15px', lineHeight: 1.55, color: '#393939' }}>
    Every chart must carry <strong>Data</strong>, <strong>Story</strong>, and <strong>Design</strong>. Reliable data without a story is noise. A story without reliable data is misleading. Both without design is forgettable.

## The three design principles

<Principle num="1" title="Simplification — remove until only the message remains">
  Strip background fills, chart borders, gridlines, legends, 3D effects, drop shadows, and any decoration that doesn't carry meaning. Round numbers (use <code>k / M / B</code>), use direct labelling instead of legends, and let whitespace do the separating.
</Principle>

<Principle num="2" title="Visual hierarchy — the most important thing must look the most important">
  Create hierarchy with **size, contrast, and position**. Big numbers before small numbers. One accent colour against neutral greys. Primary content top-left, support content bottom-right. Readers scan in a Z-path — respect it.
</Principle>

<Principle num="3" title="Layout — grid, alignment, proximity, consistency">
  Build on a 12-column grid. Align everything to the grid — no free-floating elements. Group related elements close together, separate unrelated ones with space (or a single hairline). Use the same spacing, the same type styles, the same chart styles across every page of a product. Consistency is not boring — it's what makes a reader trust you.
</Principle>

## OCHA chart styling

Every OCHA chart uses the same type, the same geometry, and a restricted palette. The defaults below are what the Humanitarian Dataviz Tool ships with.

### Typography

Use **Roboto** for titles, labels, and annotations. Use **Roboto Condensed** inside cramped chart elements (axis tick labels, small-multiple headings). Never use a serif or decorative face inside a chart.

| Element | Font | Weight | Size |
| --- | --- | --- | --- |
| Chart title | Roboto | 500 | 16–20px |
| Axis / tick labels | Roboto Condensed | 400 | 10–12px |
| Data labels | Roboto | 500 | 11–13px |
| Source line | Roboto | 400 | 10–11px, colour `#737373` |

### Geometry

- **Sharp corners on bars.** Never round bar ends — rounded corners confuse the eye about where the value actually ends.
- **Hairline gridlines only when needed**, in `--cd-grey--400` (`#E6E6E6`) at 0.5pt.
- **No drop shadows, no bevels, no gradients** on chart geometry.

### Colour — humanitarian branding styles

OCHA dataviz uses one of four branding styles depending on the product. Pick one style per product and stay in it.

Within a style, use the primary hex for the headline value and neutral greys (`--cd-grey--700` for text, `--cd-grey--400` for rules and backgrounds) for everything else. Introduce accent colours only when the data itself carries a categorical meaning (e.g. severity, sector).

### Colour — categorical data

When you must use more than one chart colour, pick from the OCHA ramp (see [Colours](?path=/docs/foundations-colours--docs)). Keep categorical charts to **five hues or fewer**. If you need more, group the tail into "Others" or switch to small multiples.

## Maps

OCHA maps follow a dedicated style so that countries and boundaries read the same way across every product.

| Element | Colour | Weight |
| --- | --- | --- |
| Ocean | `#E1E8F6` | — |
| Featured country fill | `#FFFFFF` | — |
| Neighbouring countries fill | `#E6E7E8` | — |
| Admin 0 boundary | `#737373` | 1.3pt solid |
| Admin 1 boundary | `#C7C8CA` | 0.8pt dashed |
| Coastlines | `#64BEEB` | 0.5pt |
| Capital city marker | `--brand-primary` | 4px circle |

Always cite the boundary source (e.g. "UN OCHA, CODs") and add the standard UN disclaimer on disputed territories in the footer.

## Do's and don'ts

<div style={{ margin: '16px 0 32px' }}>
  <Rule kind="do">Start bar and column charts at a **zero baseline**. Always.</Rule>
  <Rule kind="do">Include a **source line** with provider and date on every chart.</Rule>
  <Rule kind="do">**Direct-label** bars, lines, and slices. A legend is the last resort.</Rule>
  <Rule kind="do">**Round** numbers for communication. 29,762,793 → 29.8M.</Rule>
  <Rule kind="do">**Sort** categorical data by value unless the order is semantic (time, severity).</Rule>
  <Rule kind="do">Write titles that **state the finding**, not just the topic. "Funding gap widens in Horn of Africa" beats "Funding by region, 2024."</Rule>
  <Rule kind="dont">Use **3D** bars, 3D pies, drop shadows, or bevel effects.</Rule>
  <Rule kind="dont">Clip a bar-chart axis to exaggerate a difference.</Rule>
  <Rule kind="dont">Use **rainbow** colour ramps. They are hard to read and not colourblind-safe.</Rule>
  <Rule kind="dont">Use a **pie chart with more than 5 slices**, or compare multiple pies side by side.</Rule>
  <Rule kind="dont">Rotate axis labels on a vertical bar chart. Switch to horizontal instead.</Rule>
  <Rule kind="dont">Obscure a line chart with **too many point markers**. Show points only when space allows it.</Rule>
  <Rule kind="dont">Put **text over a map** unless there's a white shadow ensuring contrast.</Rule>
</div>

## Accessibility

- Every chart must have a **text alternative** — either a data table or a prose summary describing the finding.
- Colour alone never carries meaning. Pair colour with a **label, pattern, or position**.
- Minimum **4.5:1 contrast** for text, **3:1** for chart geometry against its background. Use the accessible text variants (`--brand-primary--text`, `--cd-red--aa`, etc.) for labels on white.
- Ensure charts render on **print, dark backgrounds, and screen readers** — test all three.
